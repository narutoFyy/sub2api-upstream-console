# 自己站观测与上游路由映射需求文档

## 背景

当前控制台主要关注“上游”：每个上游账号的余额、用量、倍率、模型价格、API Key 和充值能力。后续还需要把“自己的中转站”也纳入控制台观察。

这里的“自己站”指用户自己运营、对外提供 API 服务的站点。它可能接入了多个上游，也可能给不同客户、不同模型、不同渠道配置了不同的上游 API 地址和上游 Key。用户需要在一个页面里直观看到：自己的站正在连哪些上游、每条线路使用哪一个上游 Key、对应上游分组和倍率是什么。

## 核心目标

1. 新增“自己站”管理与观测页面，和现有“上游管理”分开。
2. 展示自己站接入了哪些上游 API 地址。
3. 展示自己站每条渠道、线路或模型路由使用了哪条上游 Key。
4. 将自己站的路由配置和本控制台已有的上游、上游 Key、分组倍率关联起来。
5. 让用户能快速判断某个模型请求最终会走到哪个上游、哪个 Key、哪个分组、什么倍率。

## 名词定义

| 名词 | 说明 |
| --- | --- |
| 上游 | 第三方 Sub2API / NewAPI / 兼容站点，当前控制台已支持管理 |
| 自己站 | 用户自己运营的 API 中转站或聚合站 |
| 自己站渠道 | 自己站内部的一条渠道配置，通常包含上游地址、模型、权重、状态等 |
| 上游 API | 自己站渠道实际请求的上游 base URL 或 API endpoint |
| 上游 Key | 自己站渠道调用上游时使用的密钥，可能对应本控制台已同步到的上游 Key |
| 路由映射 | 自己站渠道、模型、上游站点、上游 Key、分组之间的关联关系 |

## 需求范围

### 必须做

- 新增“自己站”配置：
  - 名称
  - 自己站地址
  - 认证方式
  - 管理 Token / 管理员账号密码
  - 备注
  - 状态

- 新增“自己站观测”页面：
  - 自己站列表
  - 单个自己站详情
  - 渠道 / 路由映射列表

- 渠道映射至少展示：
  - 自己站渠道名称或 ID
  - 渠道状态
  - 渠道模型或模型匹配规则
  - 渠道使用的上游 API 地址
  - 匹配到的本地上游站点
  - 渠道使用的上游 Key 掩码
  - 匹配到的本地上游 Key
  - 上游 Key 绑定分组
  - 分组平台
  - 分组倍率

- 当无法匹配时要明确显示原因：
  - 未匹配到上游站点
  - 未匹配到上游 Key
  - 自己站接口未返回 Key
  - 自己站接口未开放渠道读取权限

### 暂不做

- 不在第一版里自动修改自己站路由。
- 不在第一版里自动迁移 Key 或自动切换上游。
- 不破解自己站或上游的人机验证、2FA、Cloudflare 防护。
- 不展示完整上游 Key 明文，除非后续用户明确要求做“本地安全查看”并二次确认。

- 按平台筛选：OpenAI、Anthropic、Gemini、其他

## 数据匹配规则

### 匹配上游站点

优先级：

1. 自己站渠道返回的 `base_url` 与本地上游 `base_url` 完全一致。
2. 去掉末尾 `/` 后一致。
3. 域名一致且 API 前缀兼容，例如 `/api`、`/api/v1`。
4. 用户手动建立别名映射。

### 匹配上游 Key

优先级：

1. 自己站渠道返回完整 Key，掩码后与本地上游 Key 快照匹配。
2. 自己站渠道返回 Key ID，与本地上游 Key ID 匹配。
3. 自己站渠道返回 Key 前后缀，和本地掩码 Key 的前后缀匹配。
4. 用户手动绑定“该渠道使用哪个上游 Key”。

说明：如果自己站只返回加密后的 Key 或完全不返回 Key，只能显示“自己站未返回 Key”，后续需要手动绑定。

## 后端接口草案

### 自己站管理

- `GET /api/own-sites`
- `POST /api/own-sites`
- `PUT /api/own-sites/:id`
- `DELETE /api/own-sites/:id`
- `POST /api/own-sites/:id/test`
- `POST /api/own-sites/:id/sync`

### 自己站观测

- `GET /api/own-sites/:id/routes`
  - 返回该自己站的渠道 / 路由映射。

- `GET /api/own-site-routes`
  - 跨自己站聚合路由映射列表。

- `POST /api/own-site-routes/:routeId/manual-bind`
  - 手动绑定本地上游站点或上游 Key。

## 安全要求

1. 自己站凭证和上游凭证一样，必须本地加密保存。
2. 前端默认只显示掩码 Key。
3. 不把自己站接口原始响应完整暴露给前端。
4. 手动绑定属于配置变更，需要控制台登录鉴权。
5. 不绕过 Cloudflare、人机验证、2FA；如果自己站或上游要求人工验证，应提示改用 Token 模式或手动绑定。

## 需要进一步确认的问题

1. 你的“自己站”具体是 Sub2API、NewAPI，还是另一个自研项目？
2. 自己站是否有管理员接口可以列出渠道配置？
3. 自己站渠道接口里是否会返回上游完整 Key、Key ID，还是只返回掩码？
4. 自己站的一条渠道是按模型路由，还是按用户 / 分组 / 权重路由？
5. 是否需要第一版支持“手动录入路由映射”，用于自己站接口暂时拿不到完整渠道信息的情况？

## 第一版建议实现

建议第一版先做“只读观测 + 手动绑定兜底”：

1. [x] 增加自己站配置和同步入口。
2. [x] 如果自己站能读取渠道接口，就自动拉取渠道。
3. [x] 自动用 `base_url` 匹配本地上游。
4. [x] 自动用 Key ID 或掩码匹配本地上游 Key。
5. [x] 后端预留手动绑定接口。
6. [x] 前端提供“绑定上游 Key”入口，用于把自己站账号和本地上游 Key 快照关联起来。
7. [x] 页面只展示映射关系，不做自动切换和修改。

## 已实现补充

- 自己站同步优先读取 `/admin/accounts`，以账号管理中的上游账号作为观测对象。
- `上游给我们` 使用手动绑定的本地上游 Key 分组倍率，代表采购倍率。
- `我们卖出` 使用自己站账号绑定的分组倍率，代表对外售卖倍率。
- 页面显示采购倍率、售卖倍率和倍率差，便于快速发现倒挂或利润空间。

## 验收标准

- 能新增至少一个自己站。
- 能在自己站详情页看到该站的渠道 / 路由列表。
- 每条路由能显示它接入的上游 API 地址。
- 能匹配到本地上游时，显示上游名称。
- 能匹配到本地上游 Key 时，显示 Key 名称、掩码、绑定分组、平台和倍率。
- 匹配不到时，页面明确显示原因，不显示空白或误导性 `N/A`。
- 不泄露完整上游 Key 明文。
