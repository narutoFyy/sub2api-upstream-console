# UI 重构构思文档

## 背景

当前控制台功能已经比较多：上游管理、同步监控、API Key 管理、自己站观测、模型价格广场、充值、倍率变化、日志等都堆在同一个长页面里。

问题不是单个模块不好用，而是所有模块同时出现在首页，导致用户需要不断滚动和定位。信息层级混在一起后，页面会显得乱，也不容易判断“我现在应该处理什么”。

本次重构目标是先整理信息架构，再逐步调整视觉样式和交互。

## 设计目标

1. 把“一页全部展示”改成“按任务分区”。
2. 首页只展示需要立即关注的摘要和异常。
3. 每个功能模块独立成页，减少互相干扰。
4. 自己站观测重点突出进销倍率对账，不展示噪音字段。
5. 表格、筛选、详情、空状态统一风格。
6. 重构时尽量不改后端业务逻辑，先做前端布局和信息结构。

## 新信息架构

建议使用左侧导航或顶部主导航，将页面拆成以下 6 个区域。

### 1. 总览

定位：进入控制台后第一眼看到“哪里需要处理”。

展示内容：

- 上游总数
- 正常上游数
- 同步失败数量
- 低余额数量
- 今日 Token / 今日成本
- API Key 数量
- 自己站未绑定上游 Key 数量
- 采购倍率高于售卖倍率数量
- 最近异常列表

不展示内容：

- 不展示完整上游列表。
- 不展示完整 API Key 列表。
- 不展示模型价格卡片。

总览的核心是“异常优先”，不是“功能全集”。

### 2. 上游管理

定位：管理第三方上游。

主要内容：

- 上游列表
- 新增 / 编辑 / 删除上游
- 同步单个上游
- 上游余额、用量、倍率、Key 数量
- 状态筛选、标签筛选、搜索、排序

交互建议：

- 列表保持紧凑。
- 点击“详情”后打开右侧抽屉或独立详情区。
- 详情里再展示余额趋势、最近倍率、同步日志、充值入口。

### 3. API Key 管理

定位：统一查看和管理所有上游 Key。

主要内容：

- 跨上游 Key 列表
- 上游筛选
- 平台筛选
- 状态筛选
- 搜索 Key 名称 / 分组
- 创建 Key
- 启停 / 删除 Key

推荐表格列：

| 上游 | 名称 | Key | 分组 | 平台 | 分组倍率 | 状态 | 配额 | 最后使用 | 操作 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |

说明：

- API Key 管理页是上游 Key 的事实来源。
- 自己站观测里的采购倍率应该通过这里的 Key 分组倍率确认。

### 4. 自己站观测

定位：查看自己站的账号路由和进销倍率。

页面核心问题：

> 这条自己站账号接入了哪个上游 Key？上游给我们的倍率是多少？我们把它放到哪个自己站分组里卖？卖出倍率是多少？

推荐主表列：

| 自己站账号 | 接入上游 API | 匹配上游 | 绑定上游 Key | 上游给我们 | 售卖分组 | 我们卖出 | 倍率差 | 状态 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |

字段解释：

- `自己站账号`：来自自己站 `/admin/accounts` 的账号名称。
- `接入上游 API`：来自账号 `credentials.base_url`。
- `匹配上游`：本控制台已录入的上游站点。
- `绑定上游 Key`：用户手动绑定到 API Key 管理里的具体 Key。
- `上游给我们`：绑定上游 Key 的分组倍率。
- `售卖分组`：自己站账号所在分组。
- `我们卖出`：自己站账号所在分组倍率。
- `倍率差`：`我们卖出 - 上游给我们`。
- `状态`：已匹配、需绑定、倒挂、账号异常等。

不应显示：

- 不直接显示模型映射大段文本。
- 不显示原始 JSON。
- 不在主表中展示过长错误详情。

详情展开行或右侧抽屉可显示：

- 账号 ID
- 账号平台
- 账号类型
- 账号状态
- 上游 API 地址
- 自己站分组完整列表
- 账号错误信息
- 模型映射摘要
- 手动绑定记录

### 5. 模型价格广场

定位：跨上游比较模型实际价格。

主要内容：

- OpenAI 模型区
- Claude 模型区
- 每个模型的官方原价
- 每个上游叠加分组倍率后的实际价格

建议：

- 从首页移出，单独成页。
- 首页只展示“最低价提醒”或“异常价提醒”。

### 6. 日志与设置

定位：低频排查与配置。

包含：

- 同步日志
- 倍率变化记录
- 数据库备份
- 导入 / 导出
- 控制台设置

## 导航方案

优先方案：左侧导航。

结构：

```text
Sub2API Console

总览
上游管理
API Key
自己站观测
模型价格
日志设置
```

优点：

- 页面模块多，左侧导航更稳定。
- 后续新增功能不挤压顶部空间。
- 当前页面标题和操作按钮可以放在右侧顶部。

备选方案：顶部 Tabs。

适合模块数量少时使用，但当前功能已经超过 5 个，后续还会增加，顶部 Tabs 容易拥挤。

## 页面布局建议

每个页面统一为：

```text
页面标题 + 简短说明 + 主要操作按钮
筛选栏
主内容表格 / 卡片
详情抽屉 / 展开行
```

示例：

```text
自己站观测                         [新增自己站] [刷新]
查看自己站账号接入上游 Key 的采购倍率与售卖倍率。

[自己站筛选] [状态筛选] [搜索账号/上游/Key]

表格...
```

## 表格设计原则

1. 主表只展示决策字段。
2. 长文本放详情，不放主表。
3. 状态用固定词汇，不用大段句子。
4. 数字列右对齐或使用统一数字字体。
5. 危险状态突出：倒挂、未绑定、同步失败、余额不足。
6. 每行操作不超过 3 个，更多操作放菜单或详情。

## 自己站观测状态定义

| 状态 | 含义 |
| --- | --- |
| 已对账 | 已绑定上游 Key，采购倍率和售卖倍率都可计算 |
| 需绑定 | 已读取自己站账号，但还未绑定本地上游 Key |
| 倒挂 | 售卖倍率低于采购倍率 |
| 未匹配上游 | 自己站账号的上游 API 未匹配到本地上游 |
| 账号异常 | 自己站账号状态为 error / disabled 等 |

## 视觉方向

保留当前米色纸感和 serif 气质，但降低装饰感，让管理工具更清楚。

建议：

- 背景继续使用柔和暖色，但减少大面积阴影。
- 卡片圆角保留，但表格区域更规整。
- 操作按钮分清主次：新增/同步为主按钮，编辑/绑定为次按钮，删除为危险按钮。
- 统一 badge 样式：平台、状态、倍率差都用一致规则。
- 数字字段继续使用 `Times New Roman`，但只用于倍率、金额、Token 等关键数字。

## 分阶段实施

### 第一阶段：拆页面

- 新增主导航。
- 用前端状态控制当前视图。
- 将现有模块分到不同页面。
- 不改变后端接口。

验收：

- 点击导航可以切换页面。
- 总览、上游管理、API Key、自己站观测、模型价格、日志设置都能独立显示。
- 旧功能不丢失。

### 第二阶段：重做自己站观测页

- 主表改为进销倍率对账表。
- 隐藏模型映射长文本。
- 增加“需绑定 / 已对账 / 倒挂”状态。
- 绑定上游 Key 的交互更明显。

验收：

- 未绑定账号显示“需绑定”。
- 已绑定账号显示采购倍率。
- 售卖倍率来自自己站账号分组。
- 倒挂行标红。

### 第三阶段：统一表格和详情

- 抽象通用表格样式。
- 详情改为抽屉或展开区。
- 统一空状态、错误提示、加载状态。

验收：

- 各页面视觉一致。
- 页面滚动明显减少。
- 主要操作更容易找到。

## 暂不做

- 不改数据库结构，除非页面状态确实需要新增字段。
- 不做复杂路由框架。
- 不引入前端框架。
- 不改认证和同步核心逻辑。
- 不自动修改自己站路由配置。

## 待确认

1. 导航用左侧栏，还是顶部 Tabs？
2. 总览页是否保留模型价格摘要，还是完全移到模型价格页？
3. 自己站观测是否默认只展示异常和未绑定，还是展示全部账号？
4. 详情交互用右侧抽屉，还是表格行展开？
5. 是否需要移动端优先适配，还是先优化桌面端？
